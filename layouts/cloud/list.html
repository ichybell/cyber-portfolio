Phase 1: Operational Challenge

The Bottleneck: App Runner Limits

The initial architecture relied on AWS App Runner for simplicity. However, as traffic surged during peak events, hard concurrency limits caused request queuing. Coupled with a lack of granular control over instance types, this led to unpredictable billing surges up to 300%.

Phase 2: Architectural Solution

Containerizing for Control

The solution involved a full migration to **Amazon ECS on Fargate**. This allowed for custom task definitions and fine-grained auto-scaling policies based on CPU utilization and SQS queue depth.

  • check
    Spot Instance Orchestration Leveraged Fargate Spot for non-critical workloads, reducing compute costs by ~70%.
  • check
    Karpenter Implementation Replaced Cluster Autoscaler with Karpenter for just-in-time node provisioning on EKS clusters.
  • check
    Terraform IaC Modularized infrastructure code to ensure repeatable, drift-resistant environments.
main.tf — terraform
resource "aws_ecs_service" "app_service" {
  name            = "production-api"
  cluster         = aws_ecs_cluster.main.id
  task_definition = aws_ecs_task_definition.app.arn
  desired_count   = 2
  capacity_provider_strategy {
    capacity_provider = "FARGATE_SPOT"
    weight            = 80
  }
  capacity_provider_strategy {
    capacity_provider = "FARGATE"
    weight            = 20
  }
  # Auto-scaling config enabled
  # Cost optimization: 80% Spot instances
}
