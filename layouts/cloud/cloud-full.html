Migrating Celery Workers: App Runner ➜ ECS Fargate

Replatformed Celery worker pools from App Runner to ECS Fargate to gain lifecycle control, deep observability, and cost-efficient autoscaling tied to Redis queue semantics (custom metrics → CloudWatch → step scaling).

Why App Runner Didn’t Fit Celery

Lifecycle & Observability Gaps

App Runner is HTTP-oriented; Celery workers need queue-aware lifecycle, shell access, and predictable scaling. Opaque restarts dropped work, queue depth wasn’t a scaling signal, and Redis metrics weren’t available for autoscaling. We needed worker visibility, drain-aware scaling, and zero-idle cost.

visibility_off

No queue introspection

Couldn’t inspect Celery queues or worker state; hot queues hard to diagnose.

rebase_edit

Lifecycle mismatch

HTTP autoscaling ignored backlog; restarts could drop or duplicate work.

CELERY // METRICS EXPORTER
QueueLength emitted per queue → CloudWatch
Step scaling policies:
• Non-empty → +1 (ensure a worker exists)
• Backlog & drain < 300 → +1 (scale out)
• Drain > 300 → -1 (scale in)
• Empty 3m → scale to zero (save cost)
treat_missing_data tuned to avoid false scale-down

Target Architecture (ECS on Fargate)

Queue-Semantic Autoscaling

ECS services (per Celery queue set) in private subnets; Redis/Valkey on ElastiCache; a metrics exporter publishes queue depth to CloudWatch, driving Application Auto Scaling policies. Flower stays lightweight for monitoring only.

  • query_stats
    Custom CW metrics Per-queue QueueLength published every 10s; alarms feed step scaling.
  • bolt
    Scale from zero safely Non-empty → +1 ensures liveness; 3-minute empty → scale-to-zero to cut idle cost.
  • public
    Multi-country safety Country-prefixed queues; option for per-country clusters to avoid cross-traffic.
